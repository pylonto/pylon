---
title: "Monitor and manage Pylon jobs"
description: "List, inspect, stream logs, attach to, kill, and retry agent jobs. Every agent run is tracked by a short UUID and stored in a per-pylon SQLite database."
---

Every time Pylon receives a webhook or fires a cron event, it creates a **job** — a record that tracks the agent container from the moment it is queued through to completion or failure. Each job has a short UUID you can use to stream its logs, attach a shell, or forcibly stop it.

## Listing jobs

Run `pylon jobs` to see recent jobs across all pylons:

```bash
pylon jobs
```

Example output:

```
ID         PYLON                    STATUS         TRIGGERED            DURATION
a3f8c1d2   my-sentry                done           2 min ago            1m43s
09b7e4a1   my-sentry                failed         15 min ago           32s
cc21f098   weekly-audit             running        5 min ago            --
4d6a3b77   my-sentry                queued         10 sec ago           --
```

To filter jobs to a single pylon, pass its name:

```bash
pylon jobs my-sentry
```

### Job statuses

| Status | Meaning |
|--------|---------|
| `queued` | Waiting for a free agent container slot. |
| `pending` | Accepted and about to start; container is being created. |
| `running` | Agent container is active and executing. |
| `done` | Agent completed successfully. |
| `failed` | Agent exited with an error or was killed. |

## Streaming logs

Use `pylon logs` to follow the Docker output of any job, including jobs that are still running:

```bash
pylon logs a3f8c1d2
```

This streams the agent's full stdout and stderr — equivalent to `docker logs -f` on the container. Press `Ctrl+C` to stop following.

If the container has already exited, `pylon logs` prints the complete historical output before returning.

## Attaching to a running container

`pylon attach` opens an interactive shell inside a running agent container. This is useful for inspecting the workspace mid-run, reading files the agent has written, or debugging environment issues:

```bash
pylon attach cc21f098
```

The command drops you into `/bin/sh` inside the container. Type `exit` to detach without stopping the agent.

<Warning>
Attaching to a container does not pause or interrupt the agent. Any changes you make inside the container (editing files, killing processes) affect the live agent run.
</Warning>

## Killing a job

To forcibly stop a running agent container, use `pylon kill`:

```bash
pylon kill cc21f098
```

This sends `SIGKILL` to the container via `docker kill`. The job is marked `failed` in the database.

## Retrying a job

`pylon retry` is available when the daemon is running. For webhook-triggered pylons, the most reliable way to re-run a job is to re-trigger the webhook from the source service (for example, re-open or re-assign the Sentry issue). You can also use `pylon test <pylon-name>` to send a mock payload:

```bash
pylon test my-sentry
```

For cron-triggered pylons, the next scheduled run happens automatically.

## Job history

Job records are stored in a SQLite database at:

```
~/.pylon/pylons/<name>/jobs.db
```

Each pylon has its own database. The `pylon jobs` command reads from these files directly, so it works even when the daemon is not running. You can also query the database with any SQLite tool:

```bash
sqlite3 ~/.pylon/pylons/my-sentry/jobs.db "SELECT id, status, created_at FROM jobs ORDER BY created_at DESC LIMIT 10;"
```
