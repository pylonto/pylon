---
title: "Diagnose and fix common Pylon issues"
description: "Use pylon doctor to run a full health check, then work through common issues: config errors, missing images, port conflicts, webhook failures, and more."
---

When something is not working, start with `pylon doctor`. It runs a series of checks against your environment and reports each one as `ok`, `WARN`, or `FAIL` with a short explanation. Most issues it finds come with a suggested fix.

## Running the health check

```bash
pylon doctor
```

Example output:

```
Pylon Doctor

Config .............. ok    /home/you/.pylon/config.yaml found
Docker .............. ok    running (v27.3.1)
Agent image ......... ok    pylon/agent-claude:latest built
Telegram bot ........ ok    connected (@my_pylon_bot)
Telegram chat ....... ok    chat -1001234567890 accessible
Git (SSH) ........... ok    GitHub SSH key configured
OAuth session ....... ok    /home/you/.claude found
Port 8080 ........... ok    available
Pylons .............. 2 constructed (my-sentry, weekly-audit)
  my-sentry webhook ... ok    https://pylon.example.com/my-sentry reachable
systemd service ..... ok    active

All checks passed.
```

`pylon doctor` checks the following:

- **Config** — global config file exists at `~/.pylon/config.yaml`
- **Docker** — Docker daemon is running and accessible
- **Agent image** — the agent Docker image (e.g. `pylon/agent-claude`) is built
- **Telegram / Slack** — bot token is valid and the configured chat or channel is accessible
- **Git auth** — SSH key or `gh` CLI authentication is present for private repo access
- **Agent auth** — OAuth session (`~/.claude`) or API key is configured for the agent type
- **Port** — the configured port is available (or already held by Pylon itself)
- **Pylons** — lists all constructed pylons and tests webhook URL reachability for each
- **systemd** — reports whether the Pylon service is installed and active

## Common issues

<AccordionGroup>

<Accordion title="pylon start fails with 'load config: ... run pylon setup first'">
Pylon cannot find the global config file. This usually means you have not completed the initial setup.

Run `pylon setup` to create the config file interactively:

```bash
pylon setup
```

This creates `~/.pylon/config.yaml` and walks you through configuring your channel, agent type, and server settings.
</Accordion>

<Accordion title="No notification received in Telegram or Slack">
Check each of the following in order:

1. Verify the bot token is set in `~/.pylon/.env`:
   ```bash
   cat ~/.pylon/.env
   # Should contain TELEGRAM_BOT_TOKEN=... or SLACK_BOT_TOKEN=...
   ```
2. Confirm the channel type in your global config matches the token you have set:
   ```yaml
   defaults:
     channel:
       type: telegram   # or slack
   ```
3. Run `pylon doctor` — it will test the bot connection and report whether the chat or channel is accessible.
4. For Telegram: make sure the bot is an admin in the group with topic management permissions. For Slack: make sure the bot is invited to the channel.
</Accordion>

<Accordion title="Webhook not triggered / 404 from external service">
Work through this checklist:

1. Confirm `pylon start` is running and listening on the expected port.
2. Check that `server.public_url` in `~/.pylon/config.yaml` resolves to your machine. Run `pylon doctor` to test reachability automatically.
3. Verify the path configured in your external service matches the `trigger.path` in the pylon config. For example, if the pylon has `path: /my-sentry`, the webhook URL must end in `/my-sentry`.
4. If running locally, use a tunnel such as ngrok:
   ```bash
   ngrok http 8080
   ```
   Then update `server.public_url` in your config with the ngrok URL.
</Accordion>

<Accordion title="Agent container starts but fails immediately">
Stream the container logs to see the agent's output:

```bash
pylon logs <job-id>
```

Common causes:
- Missing environment variables (API keys, secrets) — check `~/.pylon/.env`
- The repo URL is wrong or the agent cannot clone it — see the "Private repo won't clone" section below
- The agent prompt references a template variable that was not populated — check the pylon config

You can also attach to the next run while it is still starting:

```bash
pylon attach <job-id>
```
</Accordion>

<Accordion title="Docker image not found / agent won't run">
The agent Docker image is built automatically during `pylon setup`. If it was not built or was removed, rebuild it:

```bash
pylon rebuild-images
```

You can verify which images are present with:

```bash
docker images pylon/agent-claude
```

If the image exists but the container still fails to start, run `pylon doctor` to confirm Docker itself is healthy.
</Accordion>

<Accordion title="Port 8080 already in use">
Another process is using the port Pylon needs. You have two options:

**Change Pylon's port.** Edit `~/.pylon/config.yaml`:

```yaml
server:
  port: 9090
```

Then restart the daemon. Remember to update `server.public_url` and any webhook URLs registered with external services to use the new port.

**Stop the other process.** Find and stop whichever process holds the port:

```bash
lsof -i :8080
kill <pid>
```
</Accordion>

<Accordion title="Private repo won't clone">
Pylon clones repos inside Docker containers using the credentials available at agent startup. To clone private repos:

1. Use the SSH URL format in your pylon config:
   ```yaml
   workspace:
     type: git-clone
     repo: git@github.com:your-org/your-repo.git
     ref: main
   ```
2. Make sure your SSH key is configured. Test it:
   ```bash
   ssh -T git@github.com
   ```
3. Alternatively, authenticate with the GitHub CLI and set up Git credential helpers:
   ```bash
   gh auth login
   gh auth setup-git
   ```

Run `pylon doctor` to confirm whether Git authentication is detected.
</Accordion>

<Accordion title="Max concurrent jobs reached / job rejected">
Pylon enforces a limit on simultaneous agent containers. When the limit is reached, new jobs are queued. If jobs are piling up faster than they complete, increase the limit in `~/.pylon/config.yaml`:

```yaml
docker:
  max_concurrent: 6
```

Then restart the daemon. Make sure your host has enough CPU and memory to support the higher concurrency before increasing this value.
</Accordion>

<Accordion title="pylon test returns 'connection refused' or fails">
`pylon test` sends a POST request to the locally running daemon. If the daemon is not running, the request fails immediately.

Start the daemon first:

```bash
pylon start
```

Then in another terminal:

```bash
pylon test my-sentry
```

If `pylon start` is running but `pylon test` still fails, check that the port in `~/.pylon/config.yaml` matches the port the daemon is listening on.
</Accordion>

</AccordionGroup>

## Config hot-reload

Pylon watches your pylon config files (`~/.pylon/pylons/<name>/pylon.yaml`) for changes and reloads them automatically. You do not need to restart the daemon after editing a pylon config. If you change the global config (`~/.pylon/config.yaml`) or add a new pylon, restart the daemon to pick up those changes.
