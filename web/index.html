How a pylon fires

Four parts wired into one pipeline.

A pylon is a named pipeline that ties a trigger, a workspace, a sandboxed agent, and a channel together. Configure it once as YAML and it runs unattended.

01 / TRIGGER
Event arrives
Webhook POST or cron fires. HMAC signature verified, payload parsed.
webhook cron HMAC-SHA256
→
02 / GATE
Human-in-the-loop
With approval: true, Slack gets Investigate / Ignore buttons. A human holds the gate.
slack telegram human-gated
→
03 / RUN
Sandboxed agent
Isolated container, workspace mounted, network scoped. The agent runs your templated prompt with payload data injected.
sandboxed git-worktree local
→
04 / RETURN
Result in chat
Output streams back to the same thread. Every job is logged — retry, attach, or kill from the CLI.
threaded reply SQLite history
Pylon construction

A pipeline is 40 lines of YAML.

No proprietary DSL. Template variables inject webhook payload fields directly into agent prompts.

pylon.yaml
CLI
.env
# ~/.pylon/pylons/sentry-triage/pylon.yaml
name: sentry-triage
description: "Triage Sentry errors"

trigger:
  type: webhook
  path: /sentry-triage
  secret: "${SENTRY_WEBHOOK_SECRET}"
  signature_header: Sentry-Hook-Signature

workspace:
  type: git-clone
  repo: "git@github.com:acme/nexus.git"
  ref: main

channel:
  type: slack
  approval: true
  topic: "{{ .body.data.event.title }}"

agent:
  type: claude
  auth: oauth
  prompt: |
    Investigate this Sentry error and suggest a fix.

    Title: {{ .body.data.event.title }}
    Culprit: {{ .body.data.event.culprit }}
    Sentry URL: {{ .body.data.event.web_url }}
  timeout: "10m"

# Construct from a template
$ pylon construct sentry-triage --from sentry

# Start the daemon
$ pylon start

What you get

Built to sit between the noisy internet and your codebase.

ISOLATION
One container per job
Every run spawns a fresh Docker container. No shared state between jobs. The agent only sees the workspace you mount, nothing else on the host.
SECRET ISOLATION
Per-pylon .env
Each pylon carries its own .env, mode 0600, never committed. Reloaded for every job. Global tokens live one level up, never leak across pipelines.
SIGNED WEBHOOKS
HMAC-SHA256 verified
Per-pylon secret + signature header. Constant-time comparison. Unsigned requests get 401. Works with Sentry out of the box.
TEMPLATED PROMPTS
Payload → prompt
Go template syntax inside prompts, topics, and messages. {{ .body.pull_request.title }} expands live for every job.
AGENT-AGNOSTIC
Claude Code or OpenCode
Reuse your local OAuth session, or bring an API key. OpenCode Zen runs with no key at all.
HOT RELOAD
Edit YAML, keep running
The daemon watches each pylon config. Change a prompt, save the file, and the next job picks it up. No restart, no dropped webhooks.
JOB HISTORY
Every run is replayable
Every run recorded with status, duration, and logs. Retry failed jobs. Attach to running containers mid-run for live debugging.
DIAGNOSTICS
Diagnose in one command
pylon doctor checks Docker, tokens, SSH, ports, reachability. Prints an ok / WARN / FAIL table so you know exactly what's wrong.
SYSTEMD READY
Run as a service
pylon service install drops a unit file, restarts on failure, and integrates with journalctl. Or just pylon start in a tmux pane.

Your server. Your keys. Your code.

Agents that touch production need adult supervision.

Pylon makes the supervision mechanical, not aspirational.

Your code never leaves the host
Single binary, Linux or macOS. No SaaS tier, no phone-home telemetry, no cloud dependency to break.
Sandboxed per job
One Docker container per run. Workspace mounted, network scoped, timeout enforced. The agent sees nothing else on the host.
Human approval on any trigger
Flip approval: true and Pylon posts Investigate / Ignore buttons. Restrict presses by user ID so only on-call can pull the trigger.
Auditable by design
Go, MIT-licensed. Signed release checksums. The whole daemon fits in the time it takes to make coffee — read every line before you trust it.
Frequently asked

Questions from the install.sh | sh crowd.

The daemon runs entirely on your server. The only outbound traffic is to the LLM provider you configure (Anthropic, OpenAI, Google) and the chat channel you pick. Updates come from GitHub Releases when you run pylon upgrade — no background phone-home.
Claude Code and OpenCode today, both with first-class integration. OpenCode Zen runs with no API key at all. Pylon treats agents as pluggable runners — the agent: block is a thin adapter, so Cursor Agent, Aider, and self-hosted models are open PRs rather than promises.
The daemon is free and MIT. Your real cost is tokens. Cap it per-job with timeout, per-pylon with max_concurrent, and globally by setting approval: true so nothing runs without a human tap. For zero-cost triage, OpenCode Zen needs no key.
Slack and Telegram today, plus stdout for local testing and a generic webhook channel. Every channel is a driver — write your own in ~200 lines of Go.
Pylon runs up to docker.max_concurrent containers in parallel (default 3, bump to whatever your host can take). A bounded queue absorbs bursts so nothing is silently dropped.
Yes. pylon start on macOS or Linux is enough. Use stdout as a channel, feed test payloads with pylon test, and point real webhook sources at your laptop with any tunnel you like (ngrok, cloudflared, tailscale funnel). No cloud account required.
